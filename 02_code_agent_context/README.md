# 02. 코드 에이전트의 동작 원리: 프로젝트와 컨텍스트 윈도우

이 문서는 실습 없이 읽는 개념 가이드입니다. 03, 04 실습에서 에이전트가 왜 그렇게 행동하는지, 왜 가끔 멍청해지는지를 이해하는 데 필요한 배경입니다.

## 1. 왜 이걸 배우나요?

코드 에이전트를 쓰다 보면 누구나 이런 경험을 합니다.

- 처음엔 잘하다가 **대화가 길어지면 앞에서 한 말을 까먹습니다.**
- 분명히 설명했는데 **엉뚱한 파일을 고칩니다.**
- 같은 요청인데 **새 대화에서 하면 더 잘합니다.**

이건 버그가 아니라 **컨텍스트 윈도우**라는 구조적 한계 때문입니다. 이 한계를 이해하면 에이전트를 훨씬 잘 부릴 수 있습니다.

## 2. Antigravity Project — 폴더가 곧 작업실

Antigravity에서 **내 컴퓨터의 폴더 하나가 프로젝트 하나**입니다. ([공식 문서](https://antigravity.google/docs/projects))

- 에이전트는 기본적으로 그 폴더 안에서만 파일을 읽고 씁니다.
- 권한, 설정, 대화 기록이 프로젝트(폴더)별로 따로 관리됩니다.
- 프론트엔드 폴더 + 백엔드 폴더처럼 여러 폴더를 하나의 프로젝트로 묶을 수도 있습니다.

비유하면 프로젝트는 **작업실**입니다. 작업실 안의 책상과 서랍(파일)은 에이전트가 자유롭게 쓰지만, 작업실 밖(내 컴퓨터의 다른 폴더)으로 나가려면 허락을 받아야 합니다.

> 실무 팁: 업무 자동화 건마다 폴더를 따로 만드세요. "보고서자동화", "설문분석"처럼 폴더를 나누면 권한도, 대화 기록도, 에이전트가 참고하는 파일도 깔끔하게 분리됩니다.

## 3. 컨텍스트 윈도우 — 에이전트의 단기 기억

### 3.1 정의

**컨텍스트 윈도우(context window)는 AI가 한 번에 기억할 수 있는 작업 기억의 크기**입니다. 지금 이 대화에서 AI가 알고 있는 모든 것 — 내 지시, AI가 읽은 파일 내용, AI 자신의 답변, 화면에 보이지 않는 내부 지침까지 — 이 전부 이 공간 안에 들어 있습니다.

크기의 단위는 **토큰(token)** 입니다. 토큰은 AI가 글을 세는 단위로, 대략 한국어 1글자 ≈ 1~2토큰, 영어 1단어 ≈ 1.3토큰 정도입니다. 모델의 컨텍스트 윈도우가 "20만 토큰"이라면 대략 책 한 권 분량이 한계라는 뜻입니다.

핵심 성질 세 가지:

1. **유한합니다.** 다 차면 오래된 내용을 요약하거나 버려야 합니다.
2. **대화가 시작되기 전부터 이미 상당 부분 차 있습니다.** (시스템 지침, 프로젝트 설정 파일, 도구 목록)
3. **화면에 안 보여도 차지합니다.** 에이전트가 "파일을 읽었습니다" 한 줄을 보여줘도, 실제로는 그 파일 전체 내용이 컨텍스트에 들어갑니다.

> 이 절의 내용은 [Claude Code 공식 문서의 "컨텍스트 윈도우 살펴보기"](https://code.claude.com/docs/ko/context-window)를 바탕으로 합니다. 페이지 상단에 대화가 진행되며 컨텍스트가 차오르는 과정을 보여주는 인터랙티브 시뮬레이션이 있으니 꼭 한번 직접 돌려보세요. Claude Code는 Anthropic의 코드 에이전트지만, 컨텍스트 동작 원리는 Antigravity를 포함한 모든 코드 에이전트에 똑같이 적용됩니다.

### 3.2 무엇이 컨텍스트를 차지하나

| 항목 | 설명 | 보이나? |
|---|---|---|
| 시스템 지침 | "너는 코딩 에이전트다, 도구는 이렇게 써라" 같은 기본 지침 | ❌ 안 보임 |
| 프로젝트 설정 파일 | 프로젝트 폴더의 `AGENTS.md` 같은 파일 (아래 3.4 참고) | ❌ 자동 로드 |
| 도구/스킬 목록 | 에이전트가 쓸 수 있는 도구들의 설명 | ❌ 안 보임 |
| 대화 내용 | 내 지시와 에이전트의 답변 | ✅ 보임 |
| **파일 읽기 결과** | 에이전트가 읽은 파일의 **전체 내용** | 한 줄 요약만 보임 |
| 명령 실행 결과 | 터미널 출력, 테스트 결과 등 전부 | 일부만 보임 |

공식 문서가 강조하는 한 문장: **"파일 읽기가 컨텍스트 사용량을 지배한다."** 에이전트가 큰 파일을 여러 개 읽는 순간 기억 공간이 순식간에 사라집니다.

### 3.3 컨텍스트가 차면 생기는 일 — 컴팩션

컨텍스트가 한계에 가까워지면 에이전트는 지금까지의 대화를 **구조화된 요약으로 교체**합니다(컴팩션, compaction). 핵심 요청, 고친 파일, 해결한 오류 같은 뼈대는 남지만, **이전에 읽었던 파일의 정확한 내용과 중간 과정은 사라집니다.** 긴 대화 끝에 에이전트가 갑자기 디테일을 까먹는 이유가 이것입니다.

### 3.4 짧게만 알아둘 개념들

다음 개념들은 코드 에이전트 생태계에서 자주 등장하는 단어들입니다. Antigravity에서도 쓰이는 개념 위주로 짧게 정리합니다.

| 개념 | 한 줄 설명 |
|---|---|
| **AGENTS.md** | 프로젝트 폴더에 두는 **에이전트용 안내문**. "이 프로젝트는 이런 규칙으로 일해라"를 적어두면 매 대화 시작 시 자동으로 읽힙니다. Antigravity도 워크스페이스의 `AGENTS.md`를 인식합니다. |
| **Skills** | 특정 작업을 잘하는 방법을 적어둔 **재사용 가능한 매뉴얼 폴더**. 필요할 때만 로드되어 컨텍스트를 아낍니다. 03에서 자세히 다룹니다. |
| **서브에이전트(sub-agent)** | 메인 에이전트가 띄우는 보조 에이전트. **자기만의 새 컨텍스트**를 갖고 일한 뒤 **요약만** 메인에게 돌려줍니다. 큰 자료 조사를 시켜도 메인의 기억이 오염되지 않는 이유입니다. Antigravity의 Dynamic Subagents가 이 개념입니다. |
| **MCP** | Model Context Protocol. 에이전트를 외부 서비스(데이터베이스, 사내 시스템 등)에 연결하는 표준 규격. "에이전트용 USB 포트"라고 생각하면 됩니다. |

### 3.5 결론: 컨텍스트는 아껴 쓰되, 의미 있게 채워라

이번 수업에서 가장 중요한 교훈입니다. **에이전트의 성능은 컨텍스트에 무엇이 들어 있느냐로 결정됩니다.** 적게 쓰면서도 유의미하게 유지하는 요령:

1. **요청을 구체적으로.** "버그 고쳐줘"보다 "`app.py`의 로그인 함수에서 나는 오류 고쳐줘"가 낫습니다. 에이전트가 뒤지는(=읽어서 컨텍스트에 넣는) 파일 수가 줄어듭니다.
2. **작업 단위로 대화를 나누기.** 한 대화에서 대시보드도 만들고 챗봇도 만들지 마세요. 새 작업은 새 대화에서.
3. **반복되는 지시는 AGENTS.md로.** 매번 "한국어로 답해, 표로 정리해"라고 말하는 대신 파일에 적어두면 모든 대화에 자동 적용됩니다.
4. **대화가 길어져 멍청해졌다 싶으면 새 대화 시작.** 필요한 맥락만 요약해서 다시 알려주는 편이 빠릅니다.
5. **큰 조사 작업은 서브에이전트에게.** "이 폴더 전체를 분석해줘" 같은 요청은 서브에이전트가 처리하고 요약만 받게 하는 것이 좋습니다.

## 4. 한 문장으로 정리

```text
프로젝트(폴더)는 에이전트의 작업실이고,
컨텍스트 윈도우는 에이전트의 단기 기억이다.
기억은 유한하므로, 좋은 지시란 "필요한 맥락만 정확히 넣어주는 지시"다.
```

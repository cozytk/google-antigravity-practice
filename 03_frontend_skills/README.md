# 03. 실습 1: frontend-design 스킬로 타이타닉 대시보드 만들기

## 목표

Anthropic의 [frontend-design 스킬](https://github.com/anthropics/skills/tree/main/skills/frontend-design)을 장착한 에이전트에게 타이타닉 승객 데이터 분석 대시보드를 만들게 합니다. 완성물은 내 컴퓨터(localhost)에서 동작하는 웹 대시보드입니다.

이 실습에서 연습하는 것은 코딩이 아니라 **에이전트와 일하는 세 가지 기술**입니다.

1. **스킬 장착** — 전문가의 작업 기준을 에이전트에게 주입하기
2. **`/grill-me`** — 만들기 전에 질의응답으로 요구사항 맞추기
3. **`/goal`** — 목표 기준으로 에이전트가 스스로 검증하며 완성도를 높이게 하기

> 시작 전에 [CONCEPT_GUIDE.md](CONCEPT_GUIDE.md)를 먼저 읽으세요.

## 실습 데이터

[`data/titanic.csv`](data/titanic.csv) — 1912년 타이타닉호 승객 891명의 공개 데이터입니다.

| 열 | 의미 |
|---|---|
| `Survived` | 생존 여부 (1=생존, 0=사망) |
| `Pclass` | 객실 등급 (1/2/3등석) |
| `Name`, `Sex`, `Age` | 이름, 성별, 나이 |
| `SibSp`, `Parch` | 동승한 형제/배우자 수, 부모/자녀 수 |
| `Fare` | 운임 요금 |
| `Embarked` | 탑승 항구 (C/Q/S) |

## 0. 준비: 프로젝트 폴더와 스킬 장착

1. 새 폴더를 만듭니다. 예: `titanic-dashboard`
2. 그 안에 `data` 폴더를 만들고 이 저장소의 `data/titanic.csv`를 복사해 넣습니다.
3. Antigravity에서 이 폴더를 프로젝트로 엽니다.
4. **스킬 장착** — 가장 바이브 코딩다운 방법은 에이전트에게 시키는 것입니다. 첫 대화로 입력:

```text
https://github.com/anthropics/skills 저장소에서 skills/frontend-design 폴더를 받아서
이 프로젝트의 .agents/skills/frontend-design 에 설치해줘.
설치 후 SKILL.md의 내용을 세 줄로 요약해서 보여줘.
```

- 이때 **네트워크 접근(read_url)과 터미널 명령 승인 카드**가 뜹니다. 01에서 배운 권한 체계가 실제로 동작하는 순간입니다. 어떤 도메인(github.com)에 무엇을 하려는 건지 읽고 승인하세요.
- 에이전트의 세 줄 요약이 CONCEPT_GUIDE에서 본 내용(템플릿 디폴트 금지, 타이포·색상 계획, 절제)과 비슷하면 성공입니다.

> 수동으로 하고 싶다면: GitHub에서 저장소를 ZIP으로 받아 `skills/frontend-design` 폴더만 프로젝트의 `.agents/skills/`에 복사해도 됩니다.

## 1. `/grill-me` — 만들기 전에 요구사항 맞추기

바로 "대시보드 만들어줘"라고 하면 에이전트는 빈칸을 전부 자기 마음대로 채웁니다. 빈칸을 **내가** 채우도록 에이전트가 나에게 질문하게 만드는 것이 `/grill-me`입니다.

대화창에 입력:

```text
/grill-me
data/titanic.csv 데이터를 분석하는 웹 대시보드를 만들려고 해.
구현을 시작하기 전에, 디자인 방향과 UI/UX에 대해 네가 결정할 수 없는 것들을 나에게 질문해.
특히 다음 항목은 반드시 나와 합의하고 넘어가:
- 대시보드의 핵심 질문 (이 화면이 답해야 하는 질문 1~3개)
- 전체적인 디자인 무드와 색
- 어떤 차트를 몇 개나 쓸지
- 필터/인터랙션이 필요한지
```

에이전트가 던지는 질문에 답하면서 요구사항을 함께 좁힙니다. 정답은 없지만, 답변 예시:

- 핵심 질문: "누가 살아남았는가 — 성별/객실 등급/나이가 생존에 어떤 영향을 줬는가"
- 무드: 1912년 시대감이 느껴지는 차분한 디자인 (frontend-design 스킬이 "주제에 뿌리내린 디자인"을 요구하므로 좋은 재료가 됩니다)
- 차트: 생존율 요약, 등급×성별 생존율, 나이 분포 등 3~5개
- 인터랙션: 객실 등급 필터 정도만

질의응답이 끝나면 에이전트에게 **합의된 내용을 요구사항 문서로 저장**하게 합니다:

```text
지금까지 합의한 내용을 REQUIREMENTS.md 파일로 정리해서 저장해줘.
```

> 왜 파일로 저장하나요? 02에서 배운 컨텍스트 윈도우 때문입니다. 대화가 길어지면 합의 내용이 요약되며 흐려질 수 있지만, 파일로 두면 언제든 다시 읽힐 수 있습니다.

## 2. 구현 시키기

```text
REQUIREMENTS.md를 기준으로 대시보드를 구현해줘.
- frontend-design 스킬을 따라 디자인할 것
- localhost에서 실행되는 웹 앱으로 만들 것
- 데이터는 data/titanic.csv를 사용할 것
구현 계획(Implementation Plan)을 먼저 보여주고 내 승인을 받은 후 진행해.
```

진행 중에 보게 될 것들:

| 보이는 것 | 의미 |
|---|---|
| Implementation Plan 아티팩트 | 계획서입니다. 차트 구성과 기술 선택을 읽고, 이상하면 코멘트로 수정 요청 후 Proceed |
| 터미널 명령 승인 카드 | 패키지 설치(`npm install` 등), 서버 실행 명령. 내용 확인 후 승인 |
| `http://localhost:3000` 같은 주소 | 완성된 대시보드 주소. 브라우저에 직접 열어보세요 |

> **localhost**는 "내 컴퓨터 자신"이라는 뜻의 주소입니다. 지금 이 대시보드는 내 컴퓨터에서만 보입니다. 인터넷에 공개하는 방법(배포)은 04에서 배웁니다.

## 3. `/goal` — 목표 기준으로 스스로 완성도 높이게 하기

첫 결과물이 나왔다면, 이제 마무리를 에이전트의 자율 루프에 맡깁니다. `/goal`의 요령은 **"무엇을 하라"가 아니라 "어떤 상태가 되면 끝인지"를 검증 가능하게 적는 것**입니다. 에이전트는 목표가 달성될 때까지 실행과 검증을 반복합니다.

현실적인 프롬프트 예시:

```text
/goal
타이타닉 대시보드를 아래 완료 조건을 전부 만족하는 상태로 만들어줘.

완료 조건:
1. localhost에서 오류 없이 실행된다 (브라우저 콘솔 에러 0건)
2. REQUIREMENTS.md의 모든 차트가 실제 데이터로 정확히 그려진다
   (예: 전체 생존율은 38.4%여야 한다 — 직접 계산해서 검증할 것)
3. 객실 등급 필터를 조작하면 모든 차트가 함께 갱신된다
4. 브라우저 창을 모바일 크기로 줄여도 레이아웃이 깨지지 않는다
5. frontend-design 스킬 기준 자기 평가: 스크린샷을 찍어 보고,
   "템플릿 디폴트로 보이는 부분"이 있으면 고친다

각 조건을 검증할 때마다 어떻게 확인했는지 보고해.
```

이 프롬프트가 효과적인 이유:

- **검증 가능한 조건** — "예쁘게"(검증 불가) 대신 "콘솔 에러 0건"(검증 가능)
- **숫자 기준 제공** — 생존율 38.4%처럼 정답을 주면 에이전트가 데이터 처리 오류를 스스로 잡습니다
- **브라우저 검증 유도** — Antigravity 에이전트는 브라우저를 직접 열어 스크린샷으로 확인할 수 있습니다. 조건 4, 5가 이 능력을 끌어냅니다
- **보고 의무** — "어떻게 확인했는지 보고해"가 있어야 대충 "됐습니다"로 끝내지 않습니다

## 5. 응용: 내 업무 데이터로

같은 절차를 그대로 내 업무에 적용할 수 있습니다.
